IntroduçãoO que é e para que serve

🎯 Propósito

O Dashboard IA360° é uma plataforma centralizada de gestão e priorização de iniciativas de automação e inteligência artificial da PGMais. Ele integra dados do Jira com métricas financeiras calculadas em tempo real, permitindo que times de liderança tomem decisões baseadas em dados — com ROI, payback e ganhos mensais visíveis para cada iniciativa.

🔄 Fluxo Principal

  • Iniciativas criadas no Jira são sincronizadas automaticamente
  • Gestores informam custos e parâmetros financeiros pelo Admin
  • O sistema calcula ROI, Payback e Ganhos em tempo real
  • A visão de Prioridade permite reordenar iniciativas via drag-and-drop
  • Relatórios visuais por Timeline (Gantt) e Entregas

👥 Usuários-alvo

  • Líderes de TI/IA: Priorização e visão estratégica
  • Gestores de Negócio: ROI e justificativas financeiras
  • Analistas de Processos: Acompanhamento de entregas
  • Administradores: Custo-hora e parâmetros por iniciativa
  • Acesso restrito a @pgmais.com.br

🧩 Painel Matriz — Cenário Atual vs Com Automação

Ao clicar em qualquer iniciativa na tabela do Dashboard, um painel detalhado é exibido entre os KPIs e a tabela. Ele apresenta uma visão comparativa dividida em duas colunas:

1 Cenário Atual

  • Tempo economizado/dia — horas liberadas por pessoa por dia (time_saved_per_day)
  • Frequência de execução — dias úteis de execução por mês (execution_days_per_month)
  • Pessoas impactadas — quantidade de colaboradores afetados (affected_people_count)
  • Custo-hora do colaborador — valor R$/hora das pessoas envolvidas (cost_per_hour)
  • Tipo de atividade — classificação da iniciativa (ex: Governança, RPA)
  • Ferramenta — tecnologia utilizada (N8N, Python, etc.)
  • Responsável — nome e avatar do responsável no Jira
  • Lead time — dias entre start_date e due_date

2 Com Automação

  • Horas economizadas/mês — total calculado: tempo/dia × dias/mês × pessoas
  • Economia mensal por tipo — separada em: horas, headcount e produtividade
  • Total ganhos mensais — soma das três fontes de ganho (destaque verde)
  • Economia anual projetada — ganhos mensais × 12
  • Investimento detalhado — custo dev, terceiros, tokens LLM, infra cloud
  • Total investimento — soma de todos os custos (destaque rosa)
  • Tempo dev estimado — horas de desenvolvimento previstas
  • ROI e Payback — cards destacados com valores calculados em tempo real
Ganhos intangíveis: Quando preenchido, um rodapé adicional exibe os ganhos qualitativos da iniciativa (ex: redução de erros, compliance, satisfação do cliente).

Cálculos & FórmulasTodos os algoritmos financeiros utilizados

Moeda: Todos os valores monetários em Reais (R$). Tempo em horas. Implementação espelhada em backend/app/initiatives/calculations.py e frontend/src/utils/calculations.js.
💰 1. Ganhos Mensais — total_gains

O ganho mensal é composto por três fontes independentes. A soma delas representa o valor que a automação gera por mês para a empresa.

1A — Ganho por Horas de Trabalho Economizadas

// Passo 1: Horas economizadas por pessoa por mês
horas_por_pessoa = time_saved_per_day × execution_days_per_month
// Ex: 1.0 h/dia × 26 dias/mês = 26 horas/mês
// Passo 2: Total de horas economizadas considerando todas as pessoas
total_horas_economizadas = horas_por_pessoa × affected_people_count
// Ex: 26 h × 1 pessoa = 26 horas
// Passo 3: Conversão para valor monetário
ganho_horas = total_horas_economizadas × cost_per_hour
// Ex: 26 h × R$36,62/h = R$950,12/mês

1B — Ganho por Redução de Headcount

ganho_headcount = headcount_reduction × monthly_employee_cost
// Ex: 0.5 FTE × R$6.000/mês = R$3.000/mês
// Aplicável quando a automação substitui função antes realizada por pessoa

1C — Ganho por Aumento de Produtividade

ganho_produtividade = productivity_increase × additional_task_value
// Ex: 200 tarefas/mês × R$5,00/tarefa = R$1.000/mês
// Aplicável quando a automação processa mais volume no mesmo tempo

Ganho Total Mensal

total_gains = ganho_horas + ganho_headcount + ganho_produtividade
// Ex: R$950,12 + R$0 + R$0 = R$950,12/mês
💸 2. Custo Total de Investimento — total_costs

O custo é composto pelo investimento em desenvolvimento mais os custos operacionais recorrentes. Representa o desembolso total para implantar e manter a automação.

2A — Custo de Desenvolvimento (interno)

horas_estimadas = development_estimate_seconds ÷ 3600
// Ex: 144.000 segundos ÷ 3.600 = 40 horas
custo_desenvolvimento = horas_estimadas × tech_hour_cost
// Ex: 40 h × R$33,64/h = R$1.345,60

2B — Custo de Terceiros (vendor/consultoria)

custo_terceiros = third_party_hours × third_party_hour_cost
// Ex: 10 h × R$150/h = R$1.500,00

2C — Custo de Tokens LLM (mensal)

custo_tokens = token_cost // valor direto do campo (R$/mês)
// Ex: R$0,22/mês (custo de API de IA generativa)

2D — Custo de Infraestrutura em Nuvem (mensal)

custo_infra = cloud_infra_cost // valor direto do campo (R$/mês)
// Ex: R$2,72/mês (N8N, Cloud Functions, etc.)

Custo Total

total_costs = custo_desenvolvimento + custo_terceiros
+ custo_tokens + custo_infra
// Ex: R$1.345,60 + R$0 + R$0,22 + R$2,72 = R$1.348,54
📈 3. ROI — roi_percent

O ROI compara o ganho mensal com o custo total de investimento. Responde: "Qual o retorno percentual da automação em relação ao investimento?"

roi_percent = ((total_gains - total_costs) / total_costs) × 100
// Exemplo numérico:
roi_percent = ((R$950,12 - R$1.348,54) / R$1.348,54) × 100
= (R$-398,42 / R$1.348,54) × 100
= -29,5% → No primeiro mês, ainda não recuperou o investimento
// Caso especial: se total_costs = 0 → ROI = None (indefinido)
// Nota: o Payback indica em quantos meses o investimento é recuperado
// e a "Economia total" nos cards mostra a projeção anual (× 12)
Interpretação: ROI positivo significa que os ganhos mensais já superam o investimento total. ROI negativo indica que a iniciativa ainda está dentro do período de payback. Use o payback para saber em quantos meses o projeto se paga.
⏱️ 4. Prazo de Retorno — payback_months

O payback indica em quantos meses o projeto gera ganhos suficientes para cobrir o custo de investimento.

payback_months = total_costs / total_gains
// Exemplo:
payback_months = R$1.348,54 / R$950,12 = 1,42 meses
// Caso especial: se total_gains = 0 → payback = None (infinito)
Interpretação: Payback de 1,42 meses significa que após 43 dias de operação, o investimento está recuperado e a automação passa a gerar lucro líquido.
🕐 5. Horas Economizadas Mensais — hours_saved

Calculado durante a sincronização com o Jira para popular o campo hours_saved:

hours_saved = time_saved_per_day × execution_days_per_month
// Ex: 1,0 h/dia × 26 dias = 26 h/mês economizadas por pessoa
📏 6. Lead Time — Visão Timeline

Calculado no frontend para exibição na visão Gantt:

lead_time_dias = due_date - start_date // diferença em dias corridos
// Ex: 30 de abril - 1 de abril = 29 dias de lead time
📊 7. Totalizadores dos Cards de KPI

Exibidos nos cards de resumo do Dashboard, agregam todas as iniciativas visíveis após filtros:

// ROI Total (soma dos ROIs individuais)
total_roi = SOMA(roi_percent para cada iniciativa com ROI definido)
// Payback Médio
payback_medio = MEDIA(payback_months para cada iniciativa com payback definido)
// Horas Economizadas por Mês (total)
total_horas = SOMA(time_saved_per_day × execution_days_per_month × affected_people_count)
// Economia Mensal Total
economia_mensal = SOMA(total_gains para todas as iniciativas)
// Economia Anual Projetada
economia_anual = economia_mensal × 12
// Horas de Desenvolvimento Estimadas
total_horas_dev = SOMA(development_estimate_seconds ÷ 3600)

SegurançaCamadas de proteção do sistema

🔑

Google OAuth 2.0 (via Supabase Auth)

Autenticação delegada ao Google. Nenhuma senha armazenada no sistema. O fluxo OAuth retorna um JWT assinado pelo Supabase, válido por sessão.

📧

Whitelist de Domínio de E-mail

Após autenticação, o sistema verifica se o e-mail termina com @pgmais.com.br. Qualquer outro domínio resulta em logout imediato. Verificação realizada tanto no frontend (AuthContext) quanto no backend (domain_check.py).

🛡️

JWT Validation no Backend

Cada requisição autenticada envia o token no header Authorization: Bearer {token}. O backend valida via GET /auth/v1/user da API do Supabase — garantindo que o token é genuíno e não expirado.

🗄️

Row-Level Security (RLS)

Habilitado na tabela initiatives. O Supabase Anon Key (usado no frontend) respeita as políticas: SELECT, INSERT e UPDATE apenas para auth.role() = 'authenticated'. DELETE não é permitido — exclusão via mudança de status.

🌐

CORS Restrito

O backend FastAPI aceita requisições somente da FRONTEND_URL configurada. Sem wildcard (*). Métodos permitidos: GET, POST, PUT, PATCH, DELETE. Headers: Authorization e Content-Type.

🔐

Separação de Chaves Supabase

Anon Key (pública, no frontend): respeita RLS. Service Key (privada, apenas no backend): bypassa RLS para operações de admin como upsert do Jira. Nunca exposta ao cliente.

✅

Validação de Dados com Pydantic v2

Todos os modelos de entrada têm constraints: campos numéricos com ge=0, tipos enforçados, comprimento máximo. Requests inválidas retornam 400 Bad Request antes de chegar ao banco.

🔒

Variáveis de Ambiente

Todas as chaves sensíveis (Supabase, Jira, Google OAuth) ficam em .env nunca commitado no repositório. O arquivo .env.example mostra a estrutura sem valores reais.

Fluxo de AutenticaçãoPasso a passo do login até a API

1

Usuário clica em "Entrar com Google"

Frontend chama supabase.auth.signInWithOAuth({ provider: 'google' }). Redireciona para página de autenticação do Google.

2

Google autentica e retorna

Google valida credenciais e retorna callback para Supabase com id_token. Supabase cria sessão e emite JWT assinado.

3

Frontend verifica domínio

AuthContext.jsx extrai o e-mail do usuário. Se não terminar em @pgmais.com.br → supabase.auth.signOut() imediato. Exibe mensagem de acesso negado.

4

JWT armazenado na sessão

Token válido salvo pelo Supabase SDK no localStorage do browser. AuthContext disponibiliza user e session para toda a aplicação.

5

Requisições autenticadas ao Backend

api.js injeta Authorization: Bearer {access_token} em todas as chamadas ao FastAPI.
